# 社区养老互助服务平台 - 功能总结

## 📋 项目概述

**项目名称**：社区养老互助服务平台 (Community Help System)

**开发框架**：SSM (Spring + SpringMVC + MyBatis)

**项目定位**：为社区老年人提供志愿者服务、关爱管理、需求发布等综合性养老互助平台

---

## 👥 用户角色体系

### 1. **管理员 (STAFF)**
- 全局管理权限
- 负责用户管理、任务分配、志愿者排班、审核等

### 2. **志愿者 (VOLUNTEER)**
- 提供志愿服务
- 管理个人排班、查看任务、记录服务历史

### 3. **家属 (FAMILY)**
- 为关爱对象发布需求
- 查看关爱计划、服务记录、志愿者信息

## 🗄️ 数据库设计

### 核心数据表

#### 1. **用户表 (sys_user)**
- 存储所有用户基本信息
- 字段：id, username, password, full_name, phone, role_type, status, created_at

#### 2. **志愿者排班表 (volunteer_schedule)** ⭐ 新增
```sql
CREATE TABLE volunteer_schedule (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    volunteer_id BIGINT NOT NULL,
    day_of_week INT NOT NULL,              -- 1-7 (周一到周日)
    time_slot VARCHAR(20) NOT NULL,        -- MORNING/AFTERNOON/EVENING
    is_available TINYINT NOT NULL,         -- 1: 可用, 0: 忙碌
    assign_source VARCHAR(20) NOT NULL,    -- VOLUNTEER: 志愿者设置, ADMIN: 管理员指派
    confirm_status VARCHAR(20) NOT NULL,   -- PENDING: 待确认, CONFIRMED: 已确认, REJECTED: 已拒绝
    reject_reason VARCHAR(255),            -- 拒绝理由
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    UNIQUE KEY uk_schedule (volunteer_id, day_of_week, time_slot)
);
```

#### 3. **任务表 (task_info)**
- 存储志愿服务任务
- 字段：id, task_title, task_type, target_id, volunteer_id, status, service_time

#### 4. **服务记录表 (service_record)**
- 记录完成的服务
- 字段：id, task_id, volunteer_id, target_id, service_time, rating, feedback

#### 5. **积分记录表 (points_record)**
- 记录积分变动
- 字段：id, user_id, points, total_points, reason, created_at

#### 6. **需求表 (demand)**
- 存储家属发布的需求
- 字段：id, family_id, demand_type, description, status, created_at

#### 7. **关爱计划表 (care_plan)**
- 存储关爱计划
- 字段：id, target_id, plan_content, frequency, responsible_volunteer_id

#### 8. **通知表 (notification)**
- 存储系统通知
- 字段：id, user_id, title, content, is_read, created_at

#### 9. **徽章表 (badge)**
- 存储徽章信息
- 字段：id, badge_code, badge_name, icon, unlock_condition

#### 10. **反馈表 (feedback)**
- 存储用户反馈
- 字段：id, user_id, task_id, rating, content, created_at

## 📊 核心业务流程

### 1. **志愿者排班完整流程** ⭐

```
┌─────────────────────────────────────────────────────────┐
│                   志愿者排班业务流程                      │
└─────────────────────────────────────────────────────────┘

【志愿者自主排班】
1. 志愿者登录 → 排班管理
2. 点击网格设置状态：空白 → 可服务 → 忙碌 → 空白
3. 点击"保存排班设置"
4. 数据库存储：assign_source = 'VOLUNTEER', confirm_status = 'CONFIRMED'

【管理员指派排班】
1. 管理员登录 → 志愿者排班管理
2. 选择志愿者，查看其当前状态（绿色=可服务，红色=忙碌）
3. 点击网格为志愿者指派时段
4. 点击"保存排班设置"
5. 数据库存储：assign_source = 'ADMIN', confirm_status = 'PENDING'
6. 该时段出现在管理员的"待志愿者确认"列表

【志愿者确认流程】
1. 志愿者登录后看到"管理员待确认排班"列表
2. 选择操作：
   - 点击"同意" → confirm_status = 'CONFIRMED', 时段生效
   - 点击"拒绝" → 弹出输入框填写理由 → confirm_status = 'REJECTED'
3. 拒绝的排班进入"已拒绝排班"列表，管理员可查看拒绝理由

【数据保护机制】
- 志愿者保存排班时，只能修改 assign_source = 'VOLUNTEER' 的记录
- 管理员指派的记录，志愿者必须通过"同意/拒绝"流程处理
- 同意后的管理员指派，会显示在志愿者的排班网格中
```

### 2. **服务任务流程**

```
需求发布 → 管理员审核 → 任务分配 → 志愿者接单 → 
服务执行 → 服务记录 → 家属评价 → 积分发放 → 徽章解锁
```

### 3. **积分系统流程**

```
服务完成 → 记录服务 → 自动计算积分 → 
更新总积分 → 刷新排名 → 检查徽章解锁条件
```

## 📈 项目亮点

### 1. **创新的三态排班设计** ⭐⭐⭐
- 不仅仅是"可用/不可用"二选一
- 增加"忙碌"状态，让管理员清楚了解志愿者的真实情况
- 避免误分配任务给忙碌的志愿者

### 2. **管理员与志愿者协同机制** ⭐⭐⭐
- 管理员可指派排班，但需志愿者确认
- 志愿者可拒绝并说明理由
- 双向沟通，避免强制分配

### 3. **数据保护与隔离** ⭐⭐
- 志愿者自主设置的排班，不会被管理员误删
- 管理员指派的排班，志愿者不能直接修改
- 清晰的数据权限边界

### 4. **右上角值班汇总表格** ⭐⭐
- 固定悬浮设计，不遮挡主内容
- 一眼掌握全局值班情况
- 鼠标悬停查看详情，无需跳转页面

### 5. **完整的激励体系**
- 积分累计
- 排行榜竞争
- 徽章解锁
- 增强志愿者参与积极性

### 6. **用户体验优化**
- 渐变色主题美观现代
- 交互式网格直观易用
- AJAX无刷新操作
- 实时反馈
